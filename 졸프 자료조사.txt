졸업 프로젝트

YOLO 레이블 객체를 따오는 건 잘하지만, 객체의 포지셔닝을 잘 따오는지 확인 필요
결론: 정답과 YOLO 예측 박스를 비교해서 수치로 평가해야 합니다.
 
1. 가장 기본: IoU(Intersection over Union)로 확인
개념: 예측 박스 vs 실제 박스 겹친 정도

✔ 판단 기준
IoU ≥ 0.5 → 맞음 (기본 기준)
IoU ≥ 0.75 → 위치 정확도 좋음
IoU ≥ 0.9 → 매우 정확

✔ 어떻게 확인?
1.	이미지 하나 선택
2.	YOLO 결과 bbox 추출
3.	정답(label) bbox 가져오기
4.	IoU 계산
-> 여러 개 평균 내면 됌

2. 중심 좌표 오차 (중요..)
IoU만 쓰면 부족함
 왜냐면 겹쳐도 중심이 틀릴 수 있기 때문!!

✔ 방법
YOLO 결과: (x_pred, y_pred)
정답: (x_gt, y_gt)
-> 거리 계산
오차 = √((x_pred - x_gt)² + (y_pred - y_gt)²)

✔ 의미
값 작을수록 위치 정확하게 나옴
실제 로봇에서는 이게 더 중요…

3. 실제로는 이렇게 많이 함
-> 2개 같이 쓰기
1.	IoU (전체 위치 정확도)
2.	중심 좌표 오차 (정밀 위치)

✔ 기준 예..
IoU 평균: 0.7 이상
중심 오차: 5~10px 이하
-> 이렇게 하면 “포지셔닝 잘 된다“라고 할 수 있다!

또한 오차가 생겼을 때 어떻게 해야할 지 관련연구 확인 필요
1. YOLO 포지셔닝 오차의 본질
YOLO는 Bounding Box 회귀(regression) 모델이라서 위치 오차는 발생할 수 밖에 없음

✔ 오차 유형
1.	위치가 살짝 밀림 (Localization Error)
2.	크기 오차 (Box Size Error)
3.	중심 좌표 오차
4.	아예 잘못 잡는 경우 (False Detection)

2. 오차 해결 방법(논문 위주)
① Loss Function 개선 (가장 중요할 듯)
-> 기존 문제
IoU 기반 loss는 겹치지 않으면 학습 안됨

✔ 해결: IoU 확장 기법
-> 대표 개념
Distance IoU
Complete IoU

✔ 중요 아이디어
IoU + 중심 거리 + 비율까지 고려

✔ 효과
위치 정확도 상승
중심 오차 감소

② Anchor Box 개선
-> 문제
Anchor가 실제 객체와 안 맞으면 위치 틀림

✔ 해결 방법
데이터셋 기반 Anchor 재설계
K-means clustering 사용

✔ 효과
초기 위치 예측 정확도 증가

③ Feature Pyramid / Multi-scale 처리
-> 문제
작은 객체에서 위치 오차 큼

✔ 해결
FPN (Feature Pyramid Network)
Multi-scale detection

✔ 효과
다양한 크기에서 정확도 향상

④ IoU-aware 모델 (고급)
개념
모델이 “이 박스가 얼마나 정확한지”까지 예측

✔ 효과
잘못된 위치 박스 제거
신뢰도 향상

⑤ Post-processing 개선
기본 YOLO 문제
NMS가 위치를 망칠 수 있음

✔ 해결
Soft-NMS
Weighted Box Fusion

✔ 효과
박스 위치 더 정확해짐

⑥ 데이터 품질 개선 (실전에서 중요)
-> 논문보다 더 중요할 수도 있음

✔ 방법
다양한 거리 / 각도 수집
정확한 라벨링
데이터 증강 (회전, 노이즈)

✔ 효과
모델 안정성 증가
오차 감소

3. ToF + YOLO 결합 (주제 )
-> YOLO 단점
2D 위치만 알 수 있음

✔ 해결
ToF활용

✔ 방법
1.	YOLO → bounding box 추출
2.	박스 영역 Depth 평균 계산
3.	실제 거리 계산

✔ 효과
3D 위치 정확도 확보
로봇 제어 가능

4. 오차 보정 알고리즘 (로봇 관점)
✔ 1) Kalman Filter
-> 핵심
Kalman Filter
이전 위치 + 현재 측정 → 보정

✔ 효과
흔들림 감소
위치 안정화

✔ 2) Sensor Fusion
-> 핵심
Sensor Fusion

✔ 조합
YOLO (객체)
ToF (거리)
LiDAR (환경

✔ 효과
오차 감소
안정성 증가

객체를 정확하게 인식, 포지셔닝 하는지 방법론 논문 제시
	1.	YOLO 모델 개선을 통한 소형 객체 탐지 성능 향상 연구
✔ 핵심 문제
YOLO는 down-sampling 과정에서
 작은 객체 위치 정보 손실 → 포지셔닝 오차 발생  

✔ 방법론
1) Anchor 재설계
데이터 기반으로 anchor box 최적화
Genetic Algorithm 사용
-> 효과
객체 위치 초기 예측 정확도 상승

2) Detection Head 구조 변경
고해상도 feature (P2) 추가
작은 객체 위치 더 잘 잡음

✔ 결과
mAP 상승 (YOLOv5 대비 개선)  
2. YOLO 기반 Feature Filter Bank를 활용한 객체 탐지

✔ 핵심 문제
YOLO는 여전히 위치 정확도 문제 존재  

✔ 방법론
1) Feature Filter Bank 추가
다양한 특징을 동시에 추출
더 풍부한 feature map 생성

2) Feature 강화
객체 경계 정보 더 잘 표현

✔ 효과
객체 위치 및 인식 정확도 향상  

3. Gaussian YOLOv3 (Localization Uncertainty 기반)

✔ 핵심 문제
YOLO는 bounding box를 “딱 하나의 값”으로 예측
→ 불확실성 표현 불가 → 위치 오차 발생

✔ 방법론
1) Bounding Box를 Gaussian으로 표현
(x, y, w, h)를 확률 분포로 모델링
의미
“이 위치가 얼마나 정확한지”까지 표현

2) Localization Uncertainty 도입
bbox 신뢰도 + 위치 정확도 같이 판단

✔ 효과
False detection 감소
위치 정확도 향상  

4. 논문 3개 공통 핵심
✔ 1. Feature 개선
더 좋은 특징 → 더 정확한 위치

✔ 2. Anchor / 구조 개선
모델 구조가 위치 정확도에 직접 영향

✔ 3. Localization 자체를 개선
단순 detection이 아니라 “정확한 위치 추정”이 핵심
Anchor 최적화를 통해 위치 예측 정확도 향상
Feature 강화로 객체 경계 정보 개선Bounding box를 확률적으로 모델링하여 오차 감소
다양한 구조 개선을 통해 localization 성능 향상

데이터 분할에서 왜 train, val, test를 나누는지 확인 필요/ 왜 train은 몇 장이고, val은 몇 장이고, test는 몇 장인지 비율 확인 필요(val은 train 영역)
1. 왜 train / val / test로 나누는가
✔ 목적
->모델의 “일반화 성능”을 확인하기 위해서

✔ 이유 설명
Train
→ 모델이 패턴을 학습
Validation
→ 학습 중간에 성능 확인
→ 하이퍼파라미터 튜닝
Test
→ 완전히 새로운 데이터로 최종 평가

✔ 핵심 문제 (안 나누면)
-> Train만 쓰면 모델이 데이터 외워버림(overfitting)
그래서
Val로 중간 점검
Test로 진짜 성능 확인

3. 왜 비율을 나누는가
✔ 핵심
-> 학습 / 검증 / 평가를 균형 있게 하기 위해

4. 데이터 비율 (정확한 기준)
✔ 대표 비율
70 : 20 : 10
80 : 10 : 10

✔ 왜 이렇게 나누냐
1) Train (가장 많음)
모델은 데이터 많이 필요

2) Validation
모델 튜닝용 → 너무 적으면 신뢰도 낮음

3) Test
최종 평가만 → 많이 필요 없음

✔ 예시
데이터 1000장일 때:
Train: 800장
Val: 100장
Test: 100장

5. 중요한 포인트
✔ 비율은 정답이 있진 않음
->데이터 상황에 따라 바뀜
데이터 많으면 → 8:1:1
데이터 적으면 → 7:2:1
매우 적으면 → K-fold
